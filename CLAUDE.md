## 项目概述

这是一个**零构建、本地优先**的个人知识库系统,特点:
- 纯静态 HTML + JavaScript,clone 即用
- 支持 Markdown / PDF / Word / 图片 / HTML 多格式预览
- 实时文件监控(SSE),本地增删文件自动刷新
- 双向链接、MOC(Map of Content)、标签系统

**技术栈**:
- 前端:原生 JS + marked.js + mermaid + pdf.js + highlight.js
- 后端:Node.js HTTP 服务(fs.watch + SSE)
- manifest 生成:PowerShell 脚本

## 文件创建规则(AI 必读)

### 决策树:新建文档放哪个目录?

#### 第一步:判断内容类型
- **多文件演示项目**(HTML + 图片 + 子页面,不可拆分的整体)→ `docs/演示项目/{项目名}/`
- **单文件 markdown 文档** → 继续第二步

#### 第二步:判断文档性质
1. **讨论技术选型/架构/工具** → `docs/技术方案/` 下选二级分组(第三步)
2. **项目总结/长对话复盘** → `docs/项目复盘/`(扁平,不分组)
3. **外部模板/测试页/示例** → `docs/参考资料/`(按用途分模板/测试)
4. **MOC(主题索引页)** → `docs/MOC-索引/`

#### 第三步:技术方案的二级分组(仅当第二步选了"技术方案")
- **AI工具/**:MCP、图片生成、AI 模型、对话分析、原型工具(Figma)
- **架构设计/**:信息架构、文件组织、系统设计、数据结构、可扩展性
- **工具流程/**:脚本、自动化、开发流程、效率工具、项目规范
- **研究调研/**:业界方案、论文、技术调研、开源项目分析

**控制指标**:
- 二级目录总数 ≤ 7 个(当前 4 个,预留 3 个扩展)
- 每个二级目录文件数 ≤ 50 篇
- 超过 50 篇时,考虑拆分或归档到 `docs/Archives/`

#### 第四步:命名规范
- **技术方案/项目复盘**:`YYYYMMDD_主题关键词.md`(8 位日期 + 下划线 + 标题)
  - 例:`20260701_Figma原型方案.md`
- **参考资料**:灵活命名,可选时间戳
  - 例:`链接功能测试页.md`、`01_方案讨论模板.md`
- **MOC**:`MOC_主题.md`
  - 例:`MOC_AI图片生成.md`、`MOC_知识库建设.md`
- **演示项目**:文件夹名无时间戳,内部结构保持原样
  - 例:`docs/演示项目/simon-demo/index.html`

### 演示项目的特殊规则
- 每个演示项目是**不可拆分的整体**,包含多个 HTML/图片/CSS/JS 文件
- **AI 不许修改演示项目内部的文件名和目录结构**(会破坏相对路径引用)
- 只能**整体移动文件夹**,不能拆分
- 新建演示项目时,在 `docs/演示项目/` 下创建独立文件夹

### 创建文档后的自检
创建文档后,简短说明选择该位置的理由(1 句话)。  
**例**:"放在 `AI工具/` 是因为讨论了 MCP 工具的使用方案"。  
如果有 20% 以上把握放错了,主动提出修正。

### 更新 MOC
新建文档后,判断是否属于已有 MOC 的主题:
- **AI 图片生成**: Pollinations、Figma MCP、图片物料对比
- **知识库建设**: 架构设计、功能规划、组织方案、可扩展性

如果属于,提醒用户"这篇可以加到 `MOC_xxx.md`"。

## MOC(Map of Content)维护规则

MOC 是"人工策展的主题入口",不是自动生成的目录。

### MOC 的作用
- 跨文件夹聚合相关文档(一篇文档可以出现在多个 MOC 里)
- 提供主题导航(比文件夹更灵活)
- 记录待深入的问题/待办事项

### MOC 的结构
```markdown
# MOC: 主题名

本 MOC 聚合所有与 XX 相关的讨论、方案、测试。

## 二级分类 1
- [[文档 A]]
- [[文档 B]] — 简短说明

## 二级分类 2
- ...

## 待深入探讨
- [ ] 问题 1
- [ ] 问题 2

---
**相关 MOC**: [[其他 MOC]]
**标签**: #tag1 #tag2
**创建日期**: YYYY-MM-DD
**维护规则**: 简述更新频率
```

### 什么时候创建新 MOC?
- 某个主题已有 **≥5 篇**相关文档
- 主题跨多个文件夹(如"AI 图片生成"涉及"技术方案/AI工具"+"演示项目/物料测试")
- 需要一个"着陆页"介绍某个主题给新人(或 3 个月后的自己)

### 什么时候更新 MOC?
- 每新建 **5 篇**相关文档,review 一次 MOC
- 项目阶段性完成(如 P0 功能开发完)
- 主题有重大变化(如技术选型调整)

## frontmatter 标准

每篇文档顶部加 YAML frontmatter(可选但推荐):

```markdown
---
created: 2026-07-01
tags: [MCP, 图片生成, 对比测试]
related: [[20260630_Figma方案]], [[物料测试]]
---

# 文档标题

正文...
```

**字段说明**:
- `created`:创建日期(可选,文件名已有时间戳)
- `tags`:扁平标签,用于筛选(不嵌套,如 `[MCP, 图片生成]` 而非 `[技术/AI/MCP]`)
- `related`:手动标注的相关文档(补充自动反向链接)

## 标签使用规范

### 什么时候添加 tags？

**必须添加**(AI 自动判断):
- 文档属于某个 MOC 主题(如"AI 图片生成"、"知识库建设")
- 跨多个文件夹的横向主题(如"MCP"散落在多个二级目录)
- 技术栈/工具类文档(如"Vue"、"React"、"Node.js"、"PowerShell")
- 项目复盘类文档(加项目名/关键技术标签)

**可以不加**:
- 临时测试页、草稿
- 演示项目(HTML/多文件项目)
- 参考资料/模板(已通过文件夹组织,无需二次分类)

### 标签命名规则

- **扁平结构**:`[MCP, 图片生成]`,不用 `[技术/AI/MCP]`(避免层级嵌套)
- **中文优先**:`图片生成` 而非 `image-generation`(提升可读性)
- **避免重复**:不用文件夹名做标签(如"技术方案"目录下的文档不加"技术方案"标签)
- **数量控制**:单篇文档标签 ≤ 5 个(超过 5 个说明分类过细)

### 常用标签库(供 AI 参考)

**工具类**:
- MCP, Figma, Pollinations, Node.js, PowerShell, Git, Claude

**主题类**:
- 图片生成, 知识库, PKM, 信息架构, 双向链接, 可扩展性

**技术类**:
- 前端, 后端, 架构设计, 性能优化, 测试, 部署

**方法论**:
- TDD, PARA, MOC, Dendron

### AI 创建文档时的 tags 判断流程

1. **判断 MOC 归属**:文档是否属于已有 MOC 主题 → 加对应标签
2. **判断技术栈**:是否涉及特定工具/语言 → 加工具标签
3. **判断横向主题**:是否跨文件夹(如多个目录都提到"MCP") → 加横向标签
4. **特殊类型**:
   - 临时文件/测试页 → 不加标签
   - 演示项目 → 不加 frontmatter
   - 参考资料 → 可选标签

**示例**:

创建一篇"MCP 图片生成工具对比"文档:
```markdown
---
created: 2026-07-01
tags: [MCP, 图片生成, Pollinations, Figma]
related: [[MOC_AI图片生成]]
---
```

创建一篇"Vue 组件设计模式"文档:
```markdown
---
created: 2026-07-01
tags: [Vue, 前端, 架构设计]
---
```

## 双向链接语法

### Wiki 链接(推荐)
```markdown
[[文档标题]]              ← 自动匹配,去时间戳前缀
[[文档标题|显示文字]]     ← 自定义显示
```

**匹配规则**(由 `app.js` 的 `resolveLink` 实现):
1. 完整路径匹配:`docs/技术方案/AI工具/xxx.md`
2. 文件名匹配:`xxx.md`
3. 标题匹配(去前缀):输入"知识库信息架构" → 匹配 `20260701113145_知识库信息架构与功能规划.md`

### 标准 Markdown 链接
```markdown
[显示文字](相对路径.md)
[显示文字](../../技术方案/AI工具/xxx.md)
```

- 相对路径会被 `processLinks` 解析为站内跳转
- 外部链接(http/https)自动开新标签

### 失效链接
如果链接的文档不存在,显示为 ⚠ 红色删除线,不可点击。

## 归档机制(防止文件膨胀)

### 触发条件
- 每季度末(Q1/Q2/Q3/Q4 结束时)
- 或活跃文档数超过 **200 篇**

### 归档规则
1. 统计最近 **3 个月没打开**的文档(通过 `mtime` 或手动判断)
2. 移动到 `docs/Archives/YYYYQN/`(如 `2026Q3/`)
3. 归档文档依然可搜索、可链接,只是不占据"活跃工作区"

### 归档后的目录结构
```
docs/
├── 技术方案/        ← 活跃文档(< 200 篇)
├── ...
└── Archives/
    ├── 2026Q3/
    │   └── (该季度归档的文档)
    └── 2026Q4/
```

## 开发规范

### 文件编辑规则
- Write/Edit 工具单次输出 **≤ 50 行**
- 超过 50 行的文件,分多次 Write + Edit 追加
- 不通过 Write/Edit 绕过行数限制(不使用 Bash 的 echo/cat 写文件)

### 功能开发优先级
- **P0**(已完成):双向链接、回退保留滚动位置
- **P1**(待开发):反向链接、正文全文搜索、标签系统、时间显示、侧边栏调宽
- **P2/P3**:文档 TOC、MOC 反向链接、关系图谱

### 测试流程
1. 语法检查:`node --check assets/js/*.js`
2. 启动服务:`node scripts/serve.js 8850`
3. 浏览器验证:打开 `http://localhost:8850`,点击链接/搜索/功能测试

## 业界对标与设计决策

### 不迁移到 Quartz(2026-07-01 决策)
**理由**:保留独特优势(零构建 + 本地全功能 + 多格式预览),Quartz 的好功能(链接/反链/搜索)可逐个自研。

### 不完全 Dendron 化(2026-07-01 决策)
**理由**:
- Dendron 的 dot 命名法对**演示项目**(HTML + 多文件)不适用
- 缺乏 Dendron 工具支持(Lookup / Refactor / Schema 验证)
- 中文 + dot 可读性下降

### 采用文件夹 + MOC + 标签混合方案(2026-07-01 决策)
**依据**:
- Miller's Law(7±2 认知负担)→ 二级分组 ≤ 7 个
- 8,000 篇笔记实战(Sébastien Dubois)→ 扁平 + MOC 可扩展到 10,000+
- Obsidian 社区共识 → 文件夹 ≤ 2 层
